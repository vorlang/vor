# Changelog

## 2026-04-07 — later
- **Raft majority check is now native Vor arithmetic** — `examples/raft.vor` and `examples/raft_cluster.vor` previously used `Vor.Examples.RaftHelpers.majority(cluster_size: cluster_size)` to decide when a candidate had won an election. The multi-agent model checker treats extern results as `:unknown` and fans out into both branches, which produced a false positive (every candidate immediately became leader regardless of vote count). The check is now `half = cluster_size / 2; if new_votes > half do ...`, which the explorer can evaluate precisely from declared parameter and integer-state types. The `majority` extern declaration was removed from both raft files. With this change `mix vor.check` on the augmented raft cluster reports `bounded: 50000 states, max depth 15` with no violation — a real (if non-exhaustive) signal that the leader-election logic is correct within bounds.
- **Parser fix: arithmetic disambiguation from Gleam paths** — `var = identifier / something` was greedily matched by the Gleam-extern path parser, which then errored out on `half = cluster_size / 2`. Both Gleam-path clauses (with and without binding) now require the token after the slash to also be an identifier so plain arithmetic falls through to the arith-binding clauses.
- **Explorer simulator: arith op atoms aligned with the IR** — `apply_arith/3` previously matched `:+ :- :* :/`, but the lowered IR uses the lexer's token names `:plus :minus :star :slash`. Both spellings are now accepted so arithmetic in handler bodies evaluates precisely instead of dropping to `:unknown`.
- **Explorer product state: instance lookup by name** — `ProductState.initial/2` was looking up the per-agent IR by `instance.type_name`, but `Vor.Explorer` builds the map keyed by `instance.name`. The lookup now tries `instance.name` first (with a `type_name` fallback). This was the root cause of the early Raft false positive — without the fix the initial agent state contained only params, so `vote_count`, `current_term`, and even `role` were missing, making every conditional fall into the `:unknown` branch.

## 2026-04-10
- **Auto-generated telemetry — Phase 1** — The compiler now emits `:telemetry.execute/3` calls in the generated Erlang code for every agent. Phase 1 covers two event types: `[:vor, :agent, :start]` (fired during init with agent name, type, and initial state) and `[:vor, :message, :received]` (fired at handler entry with message tag and agent name). The `__vor_agent_name__` field is stored in the data map during init — it defaults to the module name for standalone agents and is overridden by `__vor_name__` (system block instance name) or an explicit `__vor_agent_name__` arg. Added `:telemetry` (~> 1.0) as a dependency. Telemetry can be disabled via `config :vor, telemetry: false` (the codegen skips all telemetry calls). Both gen_server and gen_statem agents emit telemetry. All existing examples compile and run correctly.

## 2026-04-09
- **Chaos simulation — Phase 4: chaos block syntax** — System blocks now accept a `chaos do ... end` block with declarative fault injection policies: `duration Ns`, `seed N`, `kill every: MIN..MAXs`, `partition duration: MIN..MAXs`, `delay by: MIN..MAXms`, `drop probability: N` (integer percentage), `check every: Nms`, `workload rate: N`. Duration and range values accept unit suffixes (s, m, ms). The parser produces `AST.ChaosConfig` / `IR.ChaosConfig` which the simulator merges with CLI flags (CLI takes precedence). Systems without a chaos block continue to use defaults. This makes chaos scenarios version-controlled alongside the system they test.
- **Chaos simulation — Phase 3: workload generation** — `Vor.Simulator.Workload` sends protocol-matching client messages to agents at a configurable rate during simulation. Messages are built from `accepts` declarations with representative field values per type. Responses are collected; errors and timeouts are counted but don't fail the simulation (only invariant violations do). Workload events are recorded in the timeline so stats survive task shutdown. Mix task gained `--workload N` (requests per second, default 0 = disabled). Workload stats appear in the report: `workload: 287 sent/251 ok/12 err/24 timeout`. Phase 1/2 behavior unchanged when workload is 0.
- **Chaos simulation — Phase 2: message interception** — `Vor.Simulator.MessageProxy` wraps each agent in a GenServer proxy that intercepts all inter-agent traffic (casts and calls routed through the Registry). The proxy applies a configurable fault policy: `:forward` (default pass-through), `:partition` (silently drop all messages, simulating network partition), or `:delay` (hold messages for a random duration). `Vor.Simulator.SupervisorBuilder` builds a simulation supervisor tree that starts proxy-wrapped agents instead of direct agents — the proxy registers under the agent's name, the real agent is started internally without registration. The fault injector now chooses between `:kill`, `:partition`, and `:delay` based on CLI flags. Partitions are temporary — they heal after a configurable random duration. The invariant checker queries real agent state through the proxy via `MessageProxy.get_real_pid/1`. Mix task gained `--partition`, `--delay`, `--delay-min/max`, `--partition-dur-min/max`, and `--fault-interval-min/max` flags. Phase 1 kill-only behavior is unchanged when partition/delay are not enabled.
- **Chaos simulation — Phase 1** — `mix vor.simulate` starts real BEAM processes from a system block's supervisor tree, periodically kills random agents (letting OTP supervisors restart them), and checks declared system-level safety invariants against live state via `:sys.get_state/2`. Reports a timeline of kills, restarts, and invariant checks with pass/fail outcome and a reproducible seed. The invariant checker reuses the same `Vor.Explorer.Invariant` evaluator as the model checker — same invariant declaration, different state source (live process vs simulated product state). Dead processes are excluded from invariant evaluation. Mix task supports `--duration`, `--seed`, `--kill-min`, `--kill-max`, `--check-interval`, `--no-faults`, and `--verbose`. New modules: `Vor.Simulator`, `Vor.Simulator.InvariantChecker`, `Vor.Simulator.Timeline`, `Mix.Tasks.Vor.Simulate`. Raft cluster simulation passes with kill injection and supervisor recovery. Completes the three-level verification pyramid: `mix compile` (static proofs), `mix vor.check` (model checking), `mix vor.simulate` (chaos testing).

## 2026-04-07 — even later still
- **Multi-agent model checking — Phase 3: quantifiers, named agent refs, symmetry reduction** — System-level safety invariants now accept three new shapes: `never(exists A, B where A.field OP value and B.field OP value)` (with cross-agent comparisons like `A.term != B.term`), `never(exists A where A.field OP value)`, and `for_all agents, COND` (unqualified field references over each agent in turn). Named agent references — `never(n1.role == :leader)` and arbitrary boolean combinations thereof — are also supported. The `Vor.Explorer.Invariant` evaluator was rewritten to walk an operand-resolution model with bindings (`{:agent_field, var, field}` for quantifier vars, `{:named_agent_field, name, field}` for system-block instances, and `{:field, name}` for unqualified for-all references). `Vor.Explorer.Relevance` extracts referenced fields from all new shapes so abstraction stays sound. New module `Vor.Explorer.Symmetry` auto-detects when a system block is homogeneous (all instances share an agent type and have identical inbound/outbound connection counts) and the active invariants do not pin specific agent identities; in that case the BFS deduplicates against a canonical fingerprint that sorts agent states and strips sender/receiver identity from pending messages. Mix task gained `--no-symmetry` and prints `Symmetry: enabled (3 identical agents, 6× reduction)` (or `disabled`). On the augmented Raft cluster: `mix vor.check` now reaches **proven in 1001 states (vs 8008 without symmetry)** — an 8× state-space reduction. Phase 1, Phase 2, and the existing 338-test suite continue to pass unchanged; identifiers `exists` and `for_all` are recognised positionally inside system invariant bodies (NOT lexer-level keywords) so existing code that uses them as variable names still compiles.

## 2026-04-07 — even later
- **Bounded message queue (`--max-queue`)** — `Vor.Explorer.Successor` now caps the pending message buffer at `max_queue` (default 10) by truncating surplus outgoing messages from the tail. This is the lossy-network model from the multi-agent design doc and is the final piece needed for Raft Phase 2 to reach exhaustive proof: with `integer_bound: 3, max_queue: 10` the BFS now terminates in **8008 states at depth 10** with `:proven` (vs `:bounded` previously). The mix task gained `--max-queue N` and reports the cap alongside the integer bound. Increasing the queue cap recovers stronger guarantees at the cost of state-space growth.
- **Multi-agent model checking — Phase 2: state abstraction + integer saturation** — The explorer now computes per-instance relevance from each system-level invariant before BFS. A state field is *relevant* iff it is referenced in the invariant body OR appears in a guard or conditional that gates a transition to a relevant field (transitive closure with local-binding propagation). Irrelevant fields are masked with the symbolic value `:abstracted` after every successor; the simulator treats `:abstracted` exactly like `:unknown` (opaque, fans conditionals into both branches). Tracked integer fields are capped at `--integer-bound` (default 3) via post-successor saturation, so unbounded counters can no longer explode the state space. New module `Vor.Explorer.Relevance`. The mix task gained `--integer-bound N` and now prints the per-run tracked/abstracted field sets and integer bound. With Phase 2 enabled, Raft's relevance closure correctly identifies `{role, current_term, vote_count}` as tracked and `{commit_index, log, voted_for}` as abstracted; `cluster_size` and `node_id` are recognised as constant parameters and excluded from the tracked dimensions. Phase 1 callers that don't supply a relevance map continue to work unchanged (`Successor.successors/3` is still the four-arg form with `opts` defaulting to no abstraction).
- **Relevance: guard value side is now walked** — `IR.GuardExpr` carries a name on the field side and an arbitrary value expression on the value side; Phase-2 relevance now extracts state-field references from both, so guards like `when T == current_term` correctly mark `current_term` as relevant.

## 2026-04-07 — later
- **Multi-agent model checking — Phase 1** — `system` blocks now accept system-level safety invariants of the form `safety "name" proven do never(count(agents where FIELD == :VALUE) OP N) end` (operators: `>`, `>=`, `==`, `<`, `<=`). A new `mix vor.check` task runs a BFS over the reachable product state space, exploring message deliveries (pending + protocol-driven external events) and checking each state against the declared invariants. On the first violation it returns a counterexample trace; otherwise it returns either `:proven` (BFS completed) or `:bounded` (depth/state-count cap hit). Standard `mix compile` parses invariants but never runs the BFS. New modules: `Vor.Explorer`, `Vor.Explorer.ProductState`, `Vor.Explorer.Simulator`, `Vor.Explorer.Successor`, `Vor.Explorer.Invariant`, `Mix.Tasks.Vor.Check`. Successor generation drops no-op deliveries via fingerprint comparison (sorted pending queue) before they enter the visited set. The simulator interprets the existing IR action tree directly — no separate transition-table representation — and branches both ways on conditionals whose conditions data-depend on extern call results. Phase 1 scope excludes state abstraction, timeout modeling, `exists`/`for_all` quantifiers, and partial-order/symmetry reduction.

## 2026-04-07
- **Extern boundary is now an error for proven safety invariants** — A `proven` safety invariant whose verification path crosses an extern result is rejected with a compile error (`:extern_gated_invariant`). The check is path-precise: the safety verifier walks each handler in the relevant state, tracks which variables are tainted by extern call results, and reports an error only when a prohibited emit (or transition) sits inside a conditional whose condition data-depends on a tainted variable. Externs that don't influence the prohibited emit/transition (extern result used only for a transition, used unconditionally before a non-prohibited emit, in a handler whose state isn't covered by the invariant, etc.) remain fine. Consistent with the fail-closed soundness philosophy. Options when triggered: remove the extern dependency from the conditional, downgrade the invariant from `proven` to `monitored`, or restructure the handler so the prohibited emit is unreachable regardless of extern results.

## 2026-04-02
- **Lightweight type tracking** — Compiler tracks types through handler bodies and errors on guaranteed crashes: arithmetic on map, map_get on integer, list_head on map, etc. Operations on `term` (message variables, Elixir extern results) produce no diagnostics. Types inferred from state declarations, parameters, and built-in operation signatures.
- **Gleam extern support** — `extern gleam do vordb/counter.value(counter: term) :: integer end` with slash-separated module paths converted to `@`-atoms. Handler calls use same slash notation. Optional type validation against `package-interface.json`. Mixed agents with gleam + erlang + elixir externs supported.
- **Fix: variable scoping in if blocks (GAP-009)** — Variables bound inside if branches are now visible to subsequent statements in the same branch. Protocol checker threads scope sequentially through actions.
- **Named agent registration (GAP-011)** — `start_link` accepts a `name` option for `{:local, atom}`, `{:global, term}`, or `{:via, module, term}` registration. Enables multiple instances of the same agent type. Backward compatible — agents without name start anonymously.
- **Fix: post-transition extern args (GAP-007)** — Extern call arguments now read post-transition state field values. Gen_server handler body rewritten to thread data variable sequentially through transitions and extern calls.
- **Fix: Erlang module extern calls (GAP-006)** — `Erlang.erlang.system_time(unit: :millisecond)` now correctly generates `erlang:system_time(...)` instead of `'Elixir.Erlang.erlang':system_time(...)`.
- **Fix: atom literals in map ops (GAP-008)** — Atom literals (`:value`, `:key`, `:not_found`) and `nil` now work in map_get key/default positions, map_put keys, map_has keys, map_delete keys.
- **Init handlers** — `on :init do ... end` runs once at startup before accepting messages. Supports extern calls, transitions, if/else, map/list ops. No emit/send/broadcast (compile error). Only one per agent. Extern failures caught — state fields keep defaults.
- **LWW map_merge strategy** — `map_merge(store, remote, :lww)` for Last-Writer-Wins CRDTs. Compares timestamps, tiebreaks by node_id. Idempotent.
- **Native list operations** — `list_head`, `list_tail`, `list_append`, `list_prepend`, `list_length`, `list_empty`. All handle empty lists safely. Lock example updated to use native ops — zero extern calls, fully native Vor.
- **Distributed lock example** — LockManager with exclusive access, FIFO wait queue, bounded hold time via liveness monitoring, and verified safety invariant (no grant when held). Queue operations via extern helpers. 50-client stress test passes.
- **Extern calls in if/else bodies** — Extern call bindings now work inside if/else branches (needed for lock's conditional queue operations).
- **Periodic timer (`every`) construct** — `every interval_ms do ... end` for scheduled periodic actions. Uses `erlang:send_after` (fires regardless of state changes, unlike liveness timeouts). Works in gen_server and gen_statem. Supports broadcast, send, transitions, extern calls. G-Counter updated to use `every` instead of liveness hack — now a clean gen_server.
- **G-Counter CRDT example** — Three-node grow-only counter with gossip-based sync using `map_merge(counts, R, :max)`. Fully native Vor — no extern calls. Periodic broadcast via liveness monitoring. Cluster convergence verified: concurrent increments across nodes converge to correct total.
- **Native map operations** — `map_get(m, k, default)`, `map_put(m, k, v)`, `map_has(m, k)`, `map_delete(m, k)`, `map_size(m)`, `map_sum(m)`, `map_merge(m1, m2, :max)`. Work in variable bindings, transitions, emit fields, for both gen_server and gen_statem.
- **Min/max expressions** — `min(a, b)` and `max(a, b)` as built-in expressions. Work in variable bindings, transitions, and emit fields. Useful for high/low water marks and CRDT merge strategies.
- **Tutorial with verified examples** — 8-step tutorial from Echo agent to multi-agent pipeline. Every code example compiles and runs. 11 tutorial tests verify all steps.
- **Compiler trace mode** — `Vor.Compiler.compile_string(source, trace: true)` prints stage-by-stage output: source stats, token count, AST structure, lowering target, verification results, codegen info, BEAM binary size. Also available via `mix vor.compile file.vor --trace`.
- **Property-based testing** — PropCheck generates random valid Vor programs and verifies compiler invariants: compilation safety, verifier soundness, graph extraction correctness, and protocol checking consistency. 9 properties, 100 iterations each.
- **Noop keyword** — `noop` in handler bodies for intentional no-ops. Valid in cast handlers; call handlers without emit still fail completeness check.
- **List literals** — `[]`, `[1, 2, 3]`, `[:a, :b]`, `[Var1, Var2]` in emit, send, broadcast, and transition fields. Raft example updated to use real lists instead of atom placeholders.
- **Conditional transitions in gen_statem** — Transitions inside if/else branches now take effect. Each branch produces its own gen_statem return with its own data map. Data map threaded through with unique variable names. Send and broadcast read post-transition values. Resilience handlers support broadcast and send.
- **Raft consensus example** — Three-node Raft cluster using native Vor primitives: `broadcast` for vote requests, `send` with variable targets for directed responses, liveness monitoring for election timeouts, proven safety invariants. All node-to-node messaging is native Vor — externs only used for log operations.
- **Variable send targets** — `send L {:msg, fields}` where L is a pattern-bound variable, resolved at runtime via Registry lookup. Enables directed responses in protocols like Raft.
- **Guard-safe data field comparison** — Guards like `when T > current_term` use `map_get/2` (guard BIF) instead of `maps:get/2`.
- **Init state timeouts** — Gen_statem agents that start in a monitored state now get the state_timeout set at init.
- **Resilience handler full actions** — Resilience handlers can now have multiple transitions, not just a target state.

Checks against `sends` protocol declarations. Generated code iterates `__vor_connections__` and sends via the registry with graceful handling of missing peers.

- **Gen_server cast handlers** — Gen_server handlers now generate both `handle_call` and `handle_cast` clauses, so `send` (which uses `gen_server:cast`) reaches the correct handler.
- **Zero compiler warnings** — Removed dead code (`compile_conditional/3`, `gen_params_map/2`), fixed unused variables, removed unreachable clauses.

## 2026-04-01
- **Handler completeness checking** — Call handlers (those with emit) must emit on every code path. Missing else branches on if/emit blocks produce compile errors. Recursive analysis handles nested if/else. Cast handlers without else remain valid.
- **Handler coverage enforcement** — Every `accepts` in the protocol must have at least one handler. Missing handlers produce compile errors with suggestions.
- **Catch-all handler generation** — Guarded handlers automatically get catch-all clauses. Calls return `{:error, :no_matching_handler}` instead of crashing. Casts are silently ignored.
- **Bidirectional relation solver** — Relations are queryable from any direction. Equation-based relations (`fahrenheit = celsius * 9 / 5 + 32`) support compile-time symbolic inversion. Forward and inverse solve in handlers via `solve` blocks.
- **Gen_server data state** — Gen_server agents can declare mutable state fields (`state count: integer`). Fields stored in the GenServer state map with type defaults. Transition, read, and update in handlers.
- **Protocol composition checking** — `system` blocks wire agents together with `connect`. The compiler verifies that connected agents' `sends` and `accepts` have matching tags and field names. Mismatches fail compilation.
- **Multi-agent systems** — `sends` protocol declaration, `send :target {:msg, fields}` in handlers, `system` blocks with `agent` instances and `connect` topology.
- **Multiple state fields** — First enum field becomes gen_statem State, others go in Data map with type defaults.
- **Gen_statem call support** — Handlers now respond to both `cast` and `call` events. Call replies include the emitted value.
- **Richer expressions** — Standalone variable binding (`x = a + 1`), nested if/else, boolean `and`/`or` in conditions, comparison operators in guards (`>`, `<`, `>=`, `<=`), int-first arithmetic (`10 - V`).
- **Transition with expressions** — `transition count: count + 1`, `transition voted_for: C`. Data field transitions generate map updates.
- **Runtime liveness monitoring** — `monitored(within: duration)` liveness invariants generate gen_statem state timeouts that rescue stuck processes via resilience handlers.
- **Compile-time safety verification** — `proven` safety invariants are verified against the state transition graph at compile time. Violations fail compilation.
- **State graph extraction** — `mix vor.graph` prints text or Mermaid diagrams of gen_statem state machines.
- **Circuit breaker example** — Verified state machine with safety invariants: open state cannot forward requests.
- **Working rate limiter example** — Flagship example combining params, externs, conditionals, arithmetic, and invariants with ETS-backed storage.
- **If/else conditionals** — Handler bodies support `if expr do ... else ... end` with comparison operators.
- **Arithmetic expressions** — Emit fields support arithmetic (`remaining: max_requests - current`).
