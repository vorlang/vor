# Changelog

## 2026-07-12 — Raft invariant corrected: per-term leader uniqueness (Vor's first substantive multi-agent result)
- **`examples/raft_cluster.vor` now checks the *correct* property.** Phase 3a's violation of `never(count(role == :leader) > 1)` was a mis-specification, not a protocol bug: Raft guarantees at most one leader *per term* (a stale leader may briefly coexist with a newer one). The invariant is now `never(exists A, B where A.role == :leader and B.role == :leader and A.current_term == B.current_term)` — expressible with the existing pairwise-existential syntax. Cone-of-influence keeps `current_term` tracked (not abstracted), so terms are compared concretely.
- **Result: PROVEN and substantive, stable across term bounds.** `mix vor.check --max-queue 2 --integer-bound 2 --depth 40 examples/raft_cluster.vor` → `✓ Proven`, `substantive` (a leader is reachable in ~76% of states). Verified stable at `integer_bound` 2/3/4 (ruling out a term-saturation false-violation) and correctly **vacuous** under `--no-fire-timers` (no leader reachable). This is Vor's first real, non-vacuous, substantive multi-agent verification result. Evidence: `evidence/phase3a-timer-measurement.md` §7.
- **Vacuity heuristic improved for pairwise invariants.** `Invariant.subject_active?/2` for `exists A, B` now tests the condition with one agent bound to both variables (the per-entity core), so a *holding* pairwise invariant over a space where the constrained entity is reachable (e.g. leaders exist, but never two in the same term) is correctly **substantive** rather than falsely vacuous. (A purely relational condition like `A.term != B.term` is not covered by this heuristic — noted in-code.) The regression test flipped a third time — VACUOUS → VIOLATED → PROVEN — and its name records the history.

## 2026-07-12 — Phase 3a: fire timers, then measure
- **The explorer now fires timer/timeout/resilience transitions** — `Vor.Explorer.Successor` gained a third successor source alongside message-delivery and external-injection: (1) internal-trigger handlers (any `on {...}` whose tag is not an `accepts` message — monitored-liveness timeouts and bare timer atoms both lower to these — fired by injecting the trigger message, gated by the handler's own guard), and (2) periodic `every` timers (body actions run via the simulator). Timers are always-enabled nondeterministic actions (the standard model-checking treatment); a firing that produces no state change is dropped by the existing dedup, so self-loops cannot diverge. Default on; `--no-fire-timers` restores the old blind mode. Supporting fix: the simulator now evaluates `in` / `not_in` guards (so `role not_in [:candidate, :leader]` filters instead of collapsing to `:unknown`). `Vor.Explorer.Coverage` now consults the fired-handler set, so unfired-resilience/timer warnings reflect what actually fired.
- **Raft "at most one leader" is now VIOLATED** — with election reachable, `mix vor.check examples/raft_cluster.vor` finds a genuine counterexample: two leaders in *different terms* (a legal transient stale leader). The invariant as written is globally too strong — Raft guarantees one leader *per term* — so this is a mis-specified property, not a protocol bug. Found in ~0.16 s. Circuit-breaker `:half_open` and G-Counter gossip are now reachable too; lock/rate_limiter (controls) unchanged.
- **Measurement (`evidence/phase3a-timer-measurement.md`)** — the honest state space explodes with message-queue size (~8–15× per slot): exhaustive exploration is tractable at queue ≤ 3 (11k states / 0.7 s at q2, truly exhaustive; 67k / 3 s at q3; 538k / 26 s at q4) and intractable at the old reference queue 10. Bug-finding stays cheap at any config (shallow counterexample). Symmetry buys only ~2× on the honest model (vs 8× on the vacuous one). Conclusions: multi-agent checking is an opt-in deep check / bug-finder, not a compile-time operation; the interleaving explosion means partial-order reduction is the highest-value next step; the symmetry fix is marginal by comparison. New `test/features/timer_firing_test.exs`; the Raft vacuity regression flipped from VACUOUS to VIOLATED (the flip is the point).

## 2026-07-12 — Phase 1: vacuity detection as a first-class feature
- **Two-axis guarantees: strength × relevance** — every system invariant now reports not only *how strong* the evidence is (`proven` / `checked` / `monitored`) but *whether it engaged with anything at all*. A new **relevance** axis — `substantive` / `vacuous` / `unexercised` — is derived by extracting each invariant's **subject** (the atomic condition it constrains, e.g. `role == :leader`) and checking whether that subject was ever satisfiable in the explored state space. An invariant that passes over a space where its subject is never true is **vacuous** — exactly the failure that let Raft report "at most one leader, proven" over 1,001 states in which every node was a follower. New modules `Vor.Explorer.Vacuity` (subject reachability) and `Vor.Explorer.Coverage` (declared-vs-reached); new `Vor.Explorer.Invariant.subject_active?/2`.
- **Declared-but-unreached coverage** — after exploration the checker compares the declared surface against reality and reports: enum `state` values that were never reached (`role` declares `[:follower, :candidate, :leader]`, only `:follower` reached), `on {...}` handlers never entered (guard never satisfied or message never delivered), and — announcing the timer gap explicitly — `resilience` / `monitored` handlers and periodic `every` timers that never fired (with a pointer to `KNOWN_ISSUES.md §1`). Handler-firing is instrumented in the BFS via a new transient `ProductState.last_handler` tag and a `coverage: true` option on `Successor.successors/4` that returns `{successors, fired_handler_ids}`.
- **Fail-closed at the `proven` tier** — a `proven`-tier invariant that is vacuous over an *exhaustively*-explored (`:proven`) space is now an **error** (`{:error, :vacuous_proven, names, stats}`), not a silent pass — `mix vor.check` raises. Over a truncated (`:bounded`) space vacuity is a loud warning instead (the subject might appear with more exploration). New `--allow-vacuous` flag (off by default) downgrades the error for deliberate partial-model exploration. `mix vor.check` output now prints the relevance verdict and coverage warnings as prominently as the strength result. Subject-reachability is the cheap high-value special case; the general mutation-based formula vacuity (Beer et al.; Kupferman & Vardi) is noted where it would slot in but not built. New tests in `test/features/vacuity_test.exs`, including the named regression `"REGRESSION: Raft leader-uniqueness is reported VACUOUS, not a clean proof"` — the test that would have caught the original bug.

## 2026-07-12 — Phase 0: truth and cleanup
- **Fixed: identifier-routing bug in directed sends** — system-block atom params (`node_id: :node1`) were stored in agent state as the unlowered AST literal `{:atom, "node1"}` instead of the bare atom `:node1`, while agents are keyed by the bare atom. No existing example's state count changed; verified once elections are made reachable (`vote_count` now reaches a majority and a leader is elected). New regression test `test/features/directed_send_routing_test.exs`.

Gaps are informational, never errors. Builds on `Vor.Surface` (no duplicated extraction); new `Vor.Coverage` module + `Mix.Tasks.Vor.Coverage`. 15 tests.
- **`mix vor.surface` — queryable spec inventory** — parses `.vor` files (default: `lib/` + `examples/`, or `--file path` for one) and emits a structured inventory of every declaration: agents (with derived `gen_server`/`gen_statem` type, params, states), protocols (accepts/emits with fields, `where`, `max_queue`, `priority`, `defaults`), safety/liveness invariants (kind, tier, reconstructed body, source line), backpressure, externs, and systems (instances, connections, `requires`, invariants, chaos config). `--format json` (default, via Jason) is machine-readable for dashboards/CI gates; `--format text` is human-readable. Read-only — runs no verification and never touches the parser/AST. New `Vor.Surface` module + `Mix.Tasks.Vor.Surface`. 13 tests.
- **Compiler warnings cleaned up** — `mix compile --warnings-as-errors` passes again. Removed never-used optional-argument defaults on `wrap_constraint_check/7` (codegen) and `make_data_field/3` (lowering), renamed an underscored-but-used `_ret` binding, regrouped `parse_system_entries/4` clauses (moved `collect_module_segments/2` out from between them), and added the missing public `Vor.Parser.parse_system_invariant_body_public/1` that the multi-agent liveness checker referenced. No behavior change.
- **Default values on accepts fields** — `accepts {:order, customer: atom, quantity: integer default: 1}` declares a default for an optional field. When a sender omits the field, the compiled handler fills it in via `maps:merge(#{quantity => 1}, IncomingFields)` before the handler body runs; a value supplied by the sender overrides the default. `default` is a contextual keyword (still usable as an ordinary identifier elsewhere), valid only on `accepts` — defaults on `emits`/`sends` are a parse error, since the sender always includes every field. Codegen only changes handlers whose tag actually declares a default: the message map is aliased in the function head, non-defaulted fields stay exact-matched, and defaulted fields are bound from the merged map. Agents without defaults are byte-for-byte unchanged. This closes the loop on protocol compatibility: `mix vor.compat` now classifies adding a field *with* a default as COMPATIBLE and *without* a default as INCOMPATIBLE, end to end from real `.vor` files.

## 2026-04-07 — later
- **Raft majority check is now native Vor arithmetic** — `examples/raft.vor` and `examples/raft_cluster.vor` previously used `Vor.Examples.RaftHelpers.majority(cluster_size: cluster_size)` to decide when a candidate had won an election. The multi-agent model checker treats extern results as `:unknown` and fans out into both branches, which produced a false positive (every candidate immediately became leader regardless of vote count). The check is now `half = cluster_size / 2; if new_votes > half do ...`, which the explorer can evaluate precisely from declared parameter and integer-state types. The `majority` extern declaration was removed from both raft files. With this change `mix vor.check` on the augmented raft cluster reports `bounded: 50000 states, max depth 15` with no violation — a real (if non-exhaustive) signal that the leader-election logic is correct within bounds.
- **Parser fix: arithmetic disambiguation from Gleam paths** — `var = identifier / something` was greedily matched by the Gleam-extern path parser, which then errored out on `half = cluster_size / 2`. Both Gleam-path clauses (with and without binding) now require the token after the slash to also be an identifier so plain arithmetic falls through to the arith-binding clauses.
- **Explorer simulator: arith op atoms aligned with the IR** — `apply_arith/3` previously matched `:+ :- :* :/`, but the lowered IR uses the lexer's token names `:plus :minus :star :slash`. Both spellings are now accepted so arithmetic in handler bodies evaluates precisely instead of dropping to `:unknown`.
- **Explorer product state: instance lookup by name** — `ProductState.initial/2` was looking up the per-agent IR by `instance.type_name`, but `Vor.Explorer` builds the map keyed by `instance.name`. The lookup now tries `instance.name` first (with a `type_name` fallback). This was the root cause of the early Raft false positive — without the fix the initial agent state contained only params, so `vote_count`, `current_term`, and even `role` were missing, making every conditional fall into the `:unknown` branch.

## 2026-05-02
- **Protocol version compatibility checking** — `mix vor.compat new.vor --against old.vor` compares two protocol versions and reports whether a rolling deploy is safe. Detects: added/removed message types, added/removed fields (with default = compatible, without = incompatible), field type changes (widening safe, narrowing unsafe). Reports COMPATIBLE or INCOMPATIBLE verdict. New module `Vor.Compat` with `check/2` for the core comparison algorithm, `Mix.Tasks.Vor.Compat` for the CLI.
- **Backpressure declarations** — `max_queue N` at agent level or `max_queue: N` on individual `accepts` declarations. When the process mailbox exceeds the limit, non-priority messages are rejected with `{:error, {:backpressure, :queue_full}}` (calls) or silently dropped (casts) before any handler runs. `priority: true` on an `accepts` declaration bypasses backpressure — health checks and admin commands always get through. Per-message limits override agent-level limits. Generated code uses `erlang:process_info(self(), message_queue_len)` for the check. Fires `[:vor, :backpressure, :rejected]` telemetry on rejection. Agents without `max_queue` behave identically to before.
- **Parser fix: keyword-named state fields** — state declarations like `state priority: integer` now parse correctly even when the field name is a Vor keyword (e.g., `priority`, `max_queue`). The state field parser accepts both `:identifier` and `:keyword` tokens. `priority` was removed from the keyword list entirely (matched positionally in accepts options only).
- **`requires` declaration in system blocks** — `requires :app` (OTP application via `Application.ensure_all_started`) or `requires Module.Name` (module via `start_link`) declares infrastructure dependencies that `mix vor.simulate` starts before agents and stops (reverse order) after simulation. Handles `{:already_started, pid}` gracefully. Dependency failure produces a clear `{:error, :dependency_failed, target, reason}`. The model checker (`mix vor.check`) ignores `requires` entirely — it simulates on the IR without starting real processes.
- **Multi-agent liveness in `mix vor.check`** — System blocks now accept `liveness "name" proven do always(P implies eventually(Q)) end` invariants. After BFS exploration, the explorer builds an adjacency map and runs Tarjan's SCC to find cycles. For each non-trivial SCC, liveness invariants are checked: if any state in the cycle has an active obligation (P holds, Q doesn't) and no state in the cycle fulfills it (Q holds), the invariant is violated. Terminal states (no outgoing transitions or self-loops only) with active unfulfilled obligations are also detected. Liveness field references are extracted from raw body tokens so the relevance analysis correctly tracks them (preventing false negatives from field abstraction). Safety and liveness results are reported together. Systems without liveness invariants skip SCC analysis entirely (no performance impact).
- **Compile-time liveness verification** — `liveness "name" proven do always(P implies eventually(Q)) end` is now verified at compile time for single-agent gen_statem. The verifier checks: from every state where P holds but Q doesn't, is there a reachable path to a state where Q holds? If not (dead end or cycle back to non-Q states), the invariant is violated and compilation fails. Uses the existing state graph (no new exploration). New modules: `Vor.Explorer.Tarjan` (Tarjan's SCC algorithm, O(V+E)), `Vor.Explorer.LivenessChecker` (liveness body parser + single-agent reachability check). Liveness body tokens are now carried through lowering (previously discarded). `monitored` liveness is unaffected — it's still runtime-only via gen_statem state timeouts. Multi-agent liveness (product-state SCC analysis) infrastructure is in place but not yet wired into `mix vor.check`.

All four event types verified by a "full lifecycle" integration test.

This is the lossy-network model from the multi-agent design doc and is the final piece needed for Raft Phase 2 to reach exhaustive proof: with `integer_bound: 3, max_queue: 10` the BFS now terminates in **8008 states at depth 10** with `:proven` (vs `:bounded` previously). - **Multi-agent model checking — Phase 2: state abstraction + integer saturation** — The explorer now computes per-instance relevance from each system-level invariant before BFS. A state field is *relevant* iff it is referenced in the invariant body OR appears in a guard or conditional that gates a transition to a relevant field (transitive closure with local-binding propagation). Irrelevant fields are masked with the symbolic value `:abstracted` after every successor; the simulator treats `:abstracted` exactly like `:unknown` (opaque, fans conditionals into both branches). Tracked integer fields are capped at `--integer-bound` (default 3) via post-successor saturation, so unbounded counters can no longer explode the state space. New module `Vor.Explorer.Relevance`. The mix task gained `--integer-bound N` and now prints the per-run tracked/abstracted field sets and integer bound. With Phase 2 enabled, Raft's relevance closure correctly identifies `{role, current_term, vote_count}` as tracked and `{commit_index, log, voted_for}` as abstracted; `cluster_size` and `node_id` are recognised as constant parameters and excluded from the tracked dimensions. - **Relevance: guard value side is now walked** — `IR.GuardExpr` carries a name on the field side and an arbitrary value expression on the value side; Phase-2 relevance now extracts state-field references from both, so guards like `when T == current_term` correctly mark `current_term` as relevant.

## 2026-04-07
- **Extern boundary is now an error for proven safety invariants** — A `proven` safety invariant whose verification path crosses an extern result is rejected with a compile error (`:extern_gated_invariant`). The check is path-precise: the safety verifier walks each handler in the relevant state, tracks which variables are tainted by extern call results, and reports an error only when a prohibited emit (or transition) sits inside a conditional whose condition data-depends on a tainted variable. Externs that don't influence the prohibited emit/transition (extern result used only for a transition, used unconditionally before a non-prohibited emit, in a handler whose state isn't covered by the invariant, etc.) remain fine. Consistent with the fail-closed soundness philosophy. Options when triggered: remove the extern dependency from the conditional, downgrade the invariant from `proven` to `monitored`, or restructure the handler so the prohibited emit is unreachable regardless of extern results.
